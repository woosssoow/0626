# 00 · 总览

> 武侠 MUD 游戏的顶层设计。本文件只讲「是什么、做什么、按什么顺序做」，具体系统拆到各自的 `docs/NN-xxx.md`。

## 1. 游戏定位

- **题材**：武侠江湖，初入江湖的少侠从拜师、习武、闯荡到一方人物。
- **形态**：文字 MUD，多人在线，以指令 / 文本交互为核心。
- **基调**：低风险、可验证、易调参。首版不堆复杂随机，先把核心流程跑通。

## 2. 核心玩法循环（目标形态）

```
注册账号 → 创建角色 → 进入江湖
   → 拜入门派 → 修习武学 → 历练升级
   → 装备成长 → 完成任务 / 押镖 / 奇遇
   → 江湖社交 / 切磋 → 成为一方人物
```

> 首版只做到「注册 → 创建角色 → 进入江湖」并展示基础角色信息，**门派之后的统统暂不纳入**。

### 3.1 依赖链与开工顺序

**「字段已定稿」≠「可开工」**：各文档的「首版范围」指字段与规则锁定；运行时启用以下面的依赖链为准。**《战斗系统》尚未立项、模型未定，凡依赖战斗的能力一律后置，不得先行实现**（2026-07-11 定）。

```
10-运行时实体（地基，先行）
   → NPC 展示 / 纯文本问答 / 闲聊、物品与背包基础（拾取/持有/材料）、数值离线计算与面板展示

《战斗系统》（未立项）
   → NPC 战斗 / 护援 / 切磋 / 自动招式、击杀奖励与掉落、装备生效、消耗品恢复（当前气血/内力资源）

《任务系统》（未立项）  → 问答触发任务、任务物品流转
《货币 / 商店 / 交易系统》（未立项） → NPC 交易能力、物品买卖、绑定的交易限制
```

## 4. 首版 MVP 边界

**纳入**：账号注册、登录校验、角色创建（一账号一角色）、四维 / 天赋 / 福缘等基础数据、进入游戏、展示角色信息。

**暂不纳入**：门派、武学、装备、背包、货币、任务、战斗公式、社交、公会、排行榜、赛季继承。

## 5. 设计原则

- **KISS**：字段少、含义清楚、默认值明确。
- **YAGNI**：只实现当前明确需要的；不预留用不到的特性（如首版不做随机天赋、多角色、改名）。
- **绿地原则（零兼容）**：全新项目，**不写任何向后兼容、不做任何数据迁移**；按最佳实践来，发现更优解直接重构重写。详见 [99-技术架构](99-技术架构.md) §0.1。
- **可测可调**：首版用固定规则保证公平，参数集中、易于调整。
- **数据分层**：账号数据与角色数据分表保存——账号管登录安全，角色管游戏内身份与成长。

## 6. 技术取向（已定稿，详见 [99-技术架构](99-技术架构.md)）

- **驱动**：Rust（`tokio`），账号表与角色表分离。
- **脚本/热更**：Lua（`mlua`），逻辑/公式热更。
- **客户端/协议**：Flutter（GUI）经 WebSocket 承载 protobuf 连服务端；一份 `.proto` 生 Rust+Dart 两端。
- **拓扑（MVP）**：**单进程**（tokio 管 I/O + 单线程游戏主循环独占世界态与 Lua VM），网关为模块留接缝。
- **持久化**：PostgreSQL（`sqlx`）；MVP 不上 NATS/Redis，按需分期演进。
- **数据**：声明式数值用 TOML（强类型校验），公式逻辑用 Lua；二者皆热更。**不再用 CSV**。
