# 06 · 房间系统

> 房间是 MUD 世界的最小空间单位，也是玩家移动、查看、遇见 NPC、触发任务、采集、战斗和互动的基础场景。

## 1. 设计定位

房间系统负责回答三个问题：

1. 玩家现在在哪里。
2. 玩家能看到什么、遇到谁。
3. 玩家能点什么、去哪里、触发什么。

本项目是移动端为主的现代 MUD，玩家通过按钮操作，本质仍映射为 MUD 命令。因此房间不仅是地图格子，还要承载「可查看对象」「房间动作」「移动出口」「任务条件」「事件触发」等玩法入口。

## 2. 设计原则

- **房间是空间单位，不是任务脚本合集**：普通房间只记录场景、出口、对象和基础互动；复杂玩法用能力模块挂载。
- **移动端按钮优先**：玩家能点的操作应由房间系统明确给出，不依赖玩家猜命令。
- **字段少但可扩展**：首版先定清楚核心字段，特殊玩法归模块。
- **表现与规则分离**：房间描述负责氛围，限制、刷新、事件单独配置。
- **低武江湖优先**：守卫、门派规矩、城镇秩序比玄幻传送和复杂机关更重要。

## 3. 房间核心字段

每个房间都具备以下基础字段：

| 字段 | 含义 | 取值 / 示例 | 备注 |
|------|------|-------------|------|
| 房间ID | 房间唯一标识 | `shaolin_zhonglou` | 全局唯一，供出口、任务、NPC 位置引用 |
| 房间名 | 显示名称 | 钟楼 | 玩家当前位置显示 |
| 区域ID | 所属区域标识 | `shaolin` / `daohuacun` / `luoyang` | 程序引用一律走区域ID；区域中文名、区域规则归区域系统（见 §13 TODO） |
| 室内 | 是否为室内房间 | 0 / 1，默认 0 | 1 表示室内；0 表示室外 |
| 描述 | 场景描述 | 一段文字 | 玩家进入或 look 房间时展示 |
| 出口 | 可前往方向 | 见 §4 | 定义相邻房间 |
| 常驻NPC | 房间默认 NPC | NPC ID + 数量 | 具体 NPC 字段见《NPC 系统》 |
| 常驻物品 | 房间默认物品 | 物品 ID + 数量 | 普通物件、可拾取物等 |
| 可查看物件 | 场景互动物件 | 物件 ID 列表 | 物件定义归物件模板表，见 §5 |
| 房间动作 | 房间级按钮 | 见 §6 | 如打水、砍树、搜查 |
| 房间限制 | 禁止规则 | 见 §7 | 禁战、禁睡、禁快速移动等 |

## 4. 出口规则

出口表示玩家从当前房间能去哪里。

### 4.1 出口字段

| 字段 | 含义 | 示例 | 备注 |
|------|------|------|------|
| 方向 | 标准移动方向，决定九宫格挂载位 | 北 / 南 / 东 / 西 / 东北 / 西北 / 东南 / 西南 / 上 / 下 | 常规出口必填；无方向出口不填（见 §4.3）；同一方向在一个房间内只能有一个出口 |
| 目标房间 | 进入的房间ID | `shaolin_yuanmen1` | 必填 |
| 显示名称 | 移动按钮文字 | 往南走 / 向北上山 / 进屋 | 非必填；没有就显示目标房间的房间名，有则覆盖；无方向出口必填 |
| 是否显示 | 是否默认出现在按钮中 | 是 / 否 | 默认是 |
| 离开条件 | 从本房间离开前的限制 | 需完成任务 / 不可持武器 / NPC 在场阻拦 | 非必填；格式见 §8.3 |
| 失败提示 | 条件不满足时的文本 | “执法弟子拦住了你。” | 有离开条件时填写 |

### 4.2 出口方向

方向是机器字段，只有 **10 个标准取值**：

| 类型 | 方向 | UI 呈现 |
|------|------|---------|
| 平面方向 | 北 / 南 / 东 / 西 / 东北 / 西北 / 东南 / 西南 | 挂载九宫格对应格位 |
| 高低方向 | 上 / 下 | 九宫格旁两个独立按钮 |

「上山 / 下山 / 入内 / 出去 / 过桥 / 进洞」等场景文案**不是方向取值**，一律通过「显示名称」实现：出口仍占一个标准方向，按钮文字用显示名称覆盖，例如方向=北 + 显示名称「向北上山」。

### 4.3 无方向出口

不适合占用标准方向的特殊出口（如街边的「进洞」「入店」旁支），可以不填方向：

- 无方向出口**必须填显示名称**（没有方向可回退成文案）。
- 显示在九宫格之外的独立按钮区。
- 不参与地图坐标自动计算（见 §11 地图系统）。
- 其余字段（目标房间、是否显示、离开条件、失败提示）与常规出口一致。

### 4.4 出口对称规则

常规出口**必须双向对应**：A 房间的东出口指向 B，则 B 必须有西出口指向 A。

- 反向映射固定：北↔南、东↔西、东北↔西南、西北↔东南、上↔下。
- 两边各自书写（每个房间文件自包含）；内容加载时校验对称性，**漏写或方向对不上直接报错**。
- 首版**不支持单向出口**（陷阱、悬崖坠落等将来有需要再引入显式标记）。
- 无方向出口没有反方向可推，不参与对称校验；回程出口由策划自行配置，校验时仅提示不报错。

## 5. 可查看物件

可查看物件是房间里不能直接当普通物品拿走，但可以查看、点击、触发行为的场景对象。

物件采用**模板表 + 引用**的方式：物件的名称、描述、动作、条件统一定义在独立的物件模板表里，**房间只挂物件 ID 列表**（与常驻 NPC、常驻物品挂 ID 的原则一致）。通用物件（水井、蒲团、药草丛）定义一次全地图复用；独特物件（少林铜钟）单独起 ID。

物件模板字段：

| 字段 | 含义 | 示例 |
|------|------|------|
| 物件ID | 物件唯一标识 | `shuijing` / `shaolin_tongzhong` |
| 物件名 | 玩家看到的名称 | 铜钟 / 书架 / 药草丛 |
| 描述 | look 该物件时显示 | 一段文字 |
| 绑定动作 | 该物件提供的按钮 | 敲钟 / 翻找 / 采集 / 炼丹 |
| 显示条件 | 什么时候显示 | 默认显示 / 任务中显示 / 门派弟子显示 |
| 使用条件 | 什么时候可点击 | 需要道具 / 需要任务 / 需要身份 |
| 失败提示 | 不满足条件时显示 | “你不知道该如何使用。” |

示例：

| 物件 | 动作 | 用途 |
|------|------|------|
| 铜钟 | 敲钟 | 师门任务、区域广播、限时 buff |
| 药草丛 | 采集 / 施肥 | 采集、师门杂事 |
| 炼丹炉 | 炼丹 | 物品制作、任务提交 |
| 书架 | 翻找 / 整理 | 找书、答题、杂务 |
| 蒲团 | 打坐 | 领悟、恢复、剧情 |

## 6. 房间动作

房间动作是直接挂在房间上的按钮，不依赖某个具体物件。

| 字段 | 含义 | 示例 |
|------|------|------|
| 动作名 | 按钮显示文本 | 搜查 / 打水 / 砍树 |
| 动作类型 | 行为归类 | 采集 / 制作 / 搜索 / 礼拜 / 训练 / 副本 |
| 触发条件 | 点击前条件 | 任务中 / 持有道具 / 身份满足 |
| 动作时长 | 是否需要读条 | 立即 / 读条 5 秒 / 分阶段 |
| 成功结果 | 成功后发生什么 | 获得物品 / 写任务标记 / 开启出口 |
| 失败结果 | 失败后发生什么 | 无事发生 / 中毒 / 打断 / 扣除材料 |
| 冷却 | 重复使用限制 | 无 / 个人冷却 / 房间冷却 |

首版常用动作类型：

| 类型 | 例子 |
|------|------|
| 采集 | 采药、采矿、摘花、打水 |
| 劳作 | 砍柴、劈柴、打扫、擦拭、施肥、浇水 |
| 搜索 | 搜查、翻找、检查 |
| 礼拜 | 拜佛、磕头、祭拜 |
| 修行 | 打坐、领悟、练功 |
| 制作 | 炼丹、煮药、酿酒 |
| 特殊 | 放弃挑战、进入下一层、敲门 |

> 备注：可查看物件的绑定动作（§5）与房间动作在数据结构上是**同一套「互动」定义**——物件动作 = 挂在物件上的互动，房间动作 = 挂在房间上的互动。两者的条件、结果、冷却等字段完全一致，程序只维护一套结构。

## 7. 房间限制

房间限制用于规定玩家在此处不能做什么。

| 限制 | 含义 | 适用场景 |
|------|------|----------|
| 禁止战斗 | 不能主动发起战斗 | 客栈、城镇安全区、门派大殿 |
| 禁止睡觉 | 不能在此休息 | 大多数公共区域 |
| 禁止快速移动 | 不能传送 / 车马直达 | 监狱、副本、特殊剧情地 |
| 禁止坐骑 | 不能骑马进入 | 室内、寺庙、山洞 |
| 禁止武器 | 不能携带或装备武器进入 | 解剑岩、门派圣地 |
| 禁止退出副本 | 不能正常离开，只能完成或放弃 | 爬塔、试炼、剧情副本 |

限制只定义规则入口，具体扣除、战斗、传送、坐骑等细节归对应系统。

## 8. 进入与离开条件

房间的通行规则拆成进入条件和离开条件，避免所有限制都混在出口按钮里。

### 8.1 进入条件

进入目标房间前检查：

- 是否达到任务进度。
- 是否属于指定门派。
- 是否持有钥匙、令牌、信物。
- 是否满足等级、声望、好感。
- 是否处于战斗、副本、押镖等特殊状态。

### 8.2 离开条件

离开当前房间前检查：

- 玩家是否正在读条动作中。
- 是否有 NPC 阻拦。
- 是否处于强制剧情。
- 该方向是否不可通行。
- 是否携带被禁止的武器、物品或状态。

失败提示必须写清楚原因，避免玩家点按钮却不知道为什么不能走。

### 8.3 条件与结果的统一格式

本文档中所有的条件（显示条件、使用条件、触发条件、进入条件、离开条件）与结果（成功结果、失败结果）使用**同一套格式**，不各自发明写法：

- **条件** = 命名判定的引用（落地为 Lua 谓词函数），如 `quest_active("shimen_caiyao")`、`has_item("tongling")`、`is_sect("shaolin")`。
- **结果** = 命名动作的引用（落地为 Lua 动作函数），如 `give_item("yaocao", 1)`、`set_quest_flag("...")`。
- 数据（TOML）只写引用与参数；判定与执行逻辑住 Lua，二者皆可热更（声明式归 TOML、逻辑归 Lua，见技术架构文档）。
- 条件为空 = 永远通过。首版绝大多数房间不配条件，此处只定格式，不做条件 DSL。

## 9. 刷新与生成

房间可以生成 NPC 和物品，但首版先按简单规则处理。

| 类型 | 说明 |
|------|------|
| 常驻生成 | 房间刷新时生成固定 NPC / 物品 |
| 条件生成 | 满足任务、时间、身份后才生成 |
| 击败刷新 | NPC 被击败后按刷新时间重新出现 |
| 一次性生成 | 剧情或任务只生成一次 |
| 副本生成 | 按副本层数、阶段、难度动态生成 |

普通地图房间只使用常驻生成和条件生成。副本生成归「副本型房间」能力模块，不放进普通房间核心字段。

## 10. 房间类型

房间类型不是硬性职业分类，只用于策划理解和配置检查。

| 类型 | 说明 | 例子 |
|------|------|------|
| 普通地图房间 | 只负责移动、描述、常驻对象 | 山路、街道、院落 |
| 功能房间 | 提供交易、存盘、休息、车马等功能 | 客栈、药铺、驿站 |
| 任务房间 | 提供任务动作或任务条件 | 药田、书房、柴房 |
| 安全房间 | 禁战或强保护 | 城镇中心、门派大殿 |
| 限制房间 | 有进入/离开规则 | 解剑岩、监狱、山门 |
| 资源房间 | 有采集、打水、砍树等动作 | 药草丛、松树林、水井 |
| 副本房间 | 动态刷怪、记录阶段 | 试炼塔、秘境、剧情战场 |

## 11. 和其他系统的关系

| 系统 | 关系 |
|------|------|
| NPC 系统 | 房间只记录 NPC 出现在哪里；NPC 自身字段仍归 NPC 系统 |
| 任务系统 | 房间提供任务动作、进入条件、任务物件；任务流程归任务系统 |
| 物品系统 | 房间只挂物品 ID 和数量；物品属性归物品系统 |
| 战斗系统 | 房间可禁战、触发战斗场景；战斗公式归战斗系统 |
| 地图系统 | 房间、区域、出口共同组成地图；坐标由系统根据连接关系自动计算 |
| 门派系统 | 门派房间可通过身份、贡献、阶位限制进入或互动 |
| 移动端界面 | 房间提供出口按钮、房间动作按钮、可查看物件按钮 |

## 12. 初版范围

首版房间系统先设计以下能力：

- 房间核心字段。
- 区域归属。
- 出口与移动按钮。
- 常驻 NPC / 物品挂载。
- 可查看物件。
- 房间动作。
- 房间限制。
- 进入 / 离开条件。
- 简单刷新规则。

暂不深入：

- 副本房间完整规则。
- 天气、昼夜、季节影响。
- 复杂机关和多阶段解谜。
- 大规模动态事件。
- 房间归属争夺、帮派建设、地盘经营。
