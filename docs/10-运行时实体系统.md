# 10 · 运行时实体系统

> 本文认领**模板层之下的运行时状态模型**：实例身份、状态作用域、生成与销毁、地面物品、持久化边界、重启与热更策略。各系统文档（05-NPC / 06-房间 / 07-区域 / 09-物品）定义的是**模板**（TOML，只读、随内容制品发布）；模板在运行中的「活体」归本文。
>
> 战斗期状态（战斗实例、仇恨、Buff）依赖《战斗系统》——**尚未立项、模型未定**，本文只预留作用域、不定义内容（2026-07-11 定）。

## 1. 概述

05/09 一旦落地就必须回答一组模板文档没回答的问题：NPC 实例的当前气血放在哪？被击杀后谁负责按刷新时间重生？玩家丢在地上的物品归谁管、什么时候消失？重启后世界如何回到初始状态？

这些问题此前散落在各文档缝隙里（05 的刷新时间、06 §9 的生成类型、07 的刷新边界、09 §2.2 的丢弃落地），本文统一收口。

两层定义，术语全案统一：

| 层 | 载体 | 性质 |
|----|------|------|
| **模板层** | TOML（NPC / 物品 / 房间 / 区域定义） | 只读、版本化、随内容制品发布（99 §10）；加载期校验 |
| **实例层** | Rust 内存（99 §5「权威在内存」） | 运行时由模板生成的活体；除玩家资产外**不落库**，重启由模板重建 |

06 §8.3 的「出口与房间永远无状态」铁律约束的是**模板层**；实例层允许短生命周期的临时容器（在场实体、地面物品）——这不是铁律的例外，而是两层各管各的。

## 2. 设计原则

- **权威在内存**：世界实例态全部住 Rust 内存，PG 只做玩家资产存档（99 既有原则，此处业务化）。
- **重启是常规重建，不是事故**：任何时刻「模板 + 玩家存档」都能完整重建世界；因此实例态丢失无需补救、无需迁移。
- **KISS**：不上 ECS、不造通用对象框架，实体就是 Rust 结构体 + 一张实例注册表。
- **绿地零兼容**照旧：实例结构可随时重构，无迁移负担。

## 3. 实体身份：模板 ID 与实例 ID

| 标识 | 例 | 性质 |
|------|-----|------|
| 模板 ID | `daopoqiong`、`jinchuang_yao` | 内容作者定义，全局唯一，跨版本稳定；TOML / 掉落表 / 商店 / 任务引用一律用它 |
| 实例 ID | 运行时句柄（自增） | 进程内唯一；重启作废；**不出现在任何 TOML 与存档中** |

- 同一模板可同时存在多个实例（房间刷 3 个土匪 = 1 个模板 3 个实例）。
- 玩家角色是特例：持久身份以数据库主键为准，进程内另持实例句柄。
- 玩家指令定位（`look 土匪` / `kill 2.土匪`）由「同房间内模板 ID + 序号」解析到实例，规则归命令层。

## 4. 状态作用域

| 作用域 | 持有内容 | 生命周期 | 落库 |
|--------|----------|----------|:---:|
| 世界 | 实例注册表、生成点计时 | 进程 | ❌ |
| 房间实例 | 在场实体列表、地面物品容器（§6） | 进程 | ❌ |
| NPC 实例 | 当前气血 / 内力、所在房间、属性快照（08 §8）；仇恨 / 战斗态**留白待《战斗系统》** | 生成 → 销毁 | ❌ |
| 玩家角色 | 原生属性、位置、经验、背包 / 装备（含「已绑定」等物品实例标记）、任务 / 剧情标记 | 永久 | ✅ PG |
| 战斗 | **留白**——《战斗系统》定稿后补 | — | — |

## 5. 生成点：生成职责收口

现状生成规则散在三处：05 的「刷新时间」、06 §9 的生成类型、07 的区域刷新边界。收口规则：

- **生成点属于房间**：06 的「常驻 NPC / 常驻物品」挂载就是最简生成点——引用模板 ID + 数量（+ 可选刷新覆盖）。
- NPC 模板上的「刷新时间」只是**默认值**；生成点可逐处覆盖。
- **刷新计时挂在生成点上**（世界作用域），不挂在已死亡的实例上——实例销毁即彻底消失，由生成点负责再生。
- 条件生成 / 一次性生成 / 副本生成按 06 §9 的分期后置；区域仍是刷新与广播的统计边界（07）。

## 6. 地面物品（玩家丢弃落地）

执行 09 §2.2 的丢弃规则时，「落在当前房间」的宿主定义：

- 房间实例持有一个**地面容器**：纯内存、易失，不落库。
- 物品落地即挂**清理计时**（全局参数），到期自动消失；重启 / 维护即清。
- 已绑定与无价值物品按 09 §2.2 直接消失，不进地面容器。

## 7. 持久化边界

| 落库（PG） | 不落库（内存，模板重建） |
|------------|--------------------------|
| 账号、角色原生属性、位置、经验（01） | NPC 实例及其全部状态 |
| 背包与装备：物品模板 ID + 数量 + 实例标记（已绑定等） | 地面物品、生成点计时 |
| 任务 / 剧情标记（06 §8.3 的个人标记即此类） | 衍生属性（08 红线）、属性快照 |
| | 战斗态（留白） |

推论：**物品实例唯一需要持久化的状态，就是玩家资产里那份**（如绑定标记）；世界中的物品全部活在内存。

## 8. 重启、热更与内容切换

- **Lua 热更**：只替换逻辑函数表，不动实例态（99 §6 既有红线）。
- **公式 / 系数热更**：属性快照带版本、tick 边界统一失效重算（08 §8）。
- **内容版本切换**（模板变更）：走重启重建，不做在线迁移（绿地原则）；未来管理后台的「审核 → 点更新」接的也是这条路径。
- **重启序列**：读模板 → 建生成点 → 生成常驻实例 → 玩家登录按存档水合（现有登录 / 重连流程不变）。

## 9. 能力与交互的统一执行契约

06 §8.3 已锁定「命名 Lua 谓词 + 命名动作」格式；全案的能力（NPC 能力模块、物品能力块、房间动作）**统一走同一条执行通道**，不再各自发明条件 / 效果写法：

```
TOML 声明能力与参数
  → Lua 判定条件 / 做决策（只读上下文）
  → 返回受控 Effect（类型化，Rust 侧白名单）
  → Rust 校验并修改世界态
  → 推送客户端
```

新增一种能力要不要改引擎，三档边界：

| 情形 | 改动范围 |
|------|----------|
| 现有能力的新配置（新 NPC 卖新商品） | 只改 TOML |
| 用现有 Effect 原语组合的新行为 | 只改 Lua |
| 新状态、新 Effect 原语、新协议 / UI | 改引擎（Rust / proto / 客户端） |
