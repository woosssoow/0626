## 0. 一句话

**Rust 驱动 + Lua 脚本热更**的现代化多人文字 MUD：**Flutter 客户端**经 **WebSocket 承载 protobuf** 连 **单进程 Rust 服务端**（tokio 管 I/O + 单线程游戏主循环独占世界态与 Lua VM），**PostgreSQL** 落库，数值走 **TOML + Lua** 双热更。**驱动与脚本分离，一份驱动跑多服**。性能、稳定优先，对 AI 友好，按需演进不提前付分布式税。

## 0.1 绿地原则（铁律，全程适用）

- **全新项目，零历史包袱**：不写任何向后兼容、不做任何数据迁移、不留旧格式适配层。
- 设计/实现一律**按当下最佳实践**；发现更优解就**直接重构/重写**，绝不为「兼容旧代码」将就。
- 开发期（未上线）：数据库 **drop + 重建**、proto 随改随生成、存档结构随便换——怎么干净怎么来。
- **唯一**需考虑兼容的情形：**上线后**面对不可控的已发布 Flutter 客户端时的 proto 向后兼容（见 §4）；未上线前一律不管。

## 1. 选型总表（定稿）

| 维度 | 选型 | 定位 |
|------|------|------|
| 驱动语言 | **Rust**（`tokio` 异步） | 内存安全 + 无 GC 停顿 + 海量长连接 |
| 脚本/热更 | **Lua**（`mlua`，lua54/vendored） | 命令、公式、行为，文件改了即热更 |
| 客户端 | **Flutter**（Android/iOS/桌面；web 以后可选） | 一套代码多端 GUI |
| 线格式 | **protobuf (proto3)** | 一份 `.proto` 生 Rust(`prost`)+Dart 两端类型 |
| 传输 | **WebSocket 承载 protobuf 二进制帧** | 长连接双向、穿透好、弱网重连简单、web 可复用 |
| 拓扑（MVP） | **单进程**（网关为模块，留接缝） | 见 §2；拆网关进 Phase 2 |
| 权威存储 | **PostgreSQL**（`sqlx`） | 账号、角色、持久数据；账号表/角色表分离 |
| 数值-声明式 | **TOML + serde** | 四维/天赋/门派/校验规则；强类型校验当安全网 |
| 数值-公式 | **Lua** | 战斗、成长曲线、条件规则；进热更路径 |
| 密码 | **argon2** | 账号密码哈希，禁明文 |
| 日志/错误 | `tracing` / `anyhow`+`thiserror` | 结构化日志，敏感字段打码 |
| 中间件 | **MVP 不上 NATS / Redis** | 按 §12 分期，触发了再加 |

## 2. 拓扑（MVP：单进程 + 接缝）

```
   Flutter 客户端 (Android/iOS/桌面)
            │  WebSocket（protobuf 二进制帧）
   ┌────────▼─────────────────────────────────────┐
   │            Rust 驱动（单进程）                    │
   │  ┌──────────────┐  channel   ┌───────────────┐  │
   │  │ 网关层(模块)    │ ─命令(in)─▶ │  游戏主循环      │  │
   │  │ WS·鉴权·会话·  │ ◀输出(out)─ │  (单线程, 独占    │  │
   │  │ protobuf 编解码 │            │  世界态 + Lua VM) │  │
   │  └──────────────┘            └───────┬───────┘  │
   │        ▲  tokio 多线程只管 I/O            │ 异步落库/读档 │
   └────────┼────────────────────────────────┼──────┘
            │                                 ▼
       连接保活/背压                    ┌─────────────┐
                                      │ PostgreSQL   │
                                      │ 账号·角色·存档 │
                                      └─────────────┘
```

- **「网关」当前是进程内模块**，与游戏主循环只通过 `protocol` 消息 + channel 对话，**互不直碰对方内存**——这就是「接缝」。Phase 2 拆进程时把它挪成独立二进制、channel 换网络传输即可，领域逻辑与消息契约原样搬。
- **核心原则（LPMud 思想）：状态住驱动（Rust/内存），行为住脚本（Lua）。** 热更换逻辑不丢世界态——这是热更能安全做的根基。

## 3. 运行时模型（最关键，别跑偏）

`mlua` 的 Lua VM **单线程**，`tokio` 多线程。二者这样接：

```
WS 连接(tokio 多线程, 只做 I/O) ──解码 protobuf→ bounded channel ──▶ 游戏主循环(单线程)
                                                                      │ 独占 World + Lua VM
        ◀── 编码 protobuf ── bounded channel ◀── 输出帧 ◀────────────┘
```

- **所有游戏逻辑 + Lua 调用全压到一个「游戏主循环」线程串行跑**：天然无数据竞争、确定性强、Lua 单线程刚好契合、热更安全。
- **tokio 只管异步 I/O**（WS 收发、DB I/O、定时器、密码哈希丢 blocking pool），**绝不阻塞主循环**。
- **Lua 红线**：脚本里不许 await、不许阻塞 I/O、不许直接碰共享 Rust 状态。
- **背压**：每连接输出队列设上限，慢客户端背压/断开，**别无限排队拖垮全服**（MUD 最易死在慢客户端）。
- **tick 心跳**：固定 **100ms**（战斗等可另设更细），跑气血/内力恢复、NPC AI、定时事件、autosave；MVP 以事件驱动为主，tick 兜定时任务。Lua 通过 `heartbeat()`/定时器接口挂钩。

## 4. 客户端协议（Flutter + protobuf + WebSocket）

- **消息信封**：每帧一个 `Envelope`，最少字段 `oneof payload` + `seq` + `ack`，可选 `request_id` / `timestamp` / `client_version`。单连接多路复用所有消息类型。
- **鉴权**：握手只建连接；**第一条业务帧 `AuthRequest`**（账号+token）→ 成功后绑定账号与角色。**绝不把 token 放 URL**（会进日志/代理/历史）。
- **心跳**：服务端 ping / 客户端 pong，**15–30s**，连续 **2–3 次**无响应才断；断的是连接，不立刻销毁角色。
- **断线重连状态机**：短断线（手机切前后台、WiFi↔流量）**保留 session 60–180s**；重连用 token **接管原 session**，宽限期内重挂回还在场的角色，**别每次都重走登录→进世界全流程**；超时后角色离世或进离线保护。
- **服务端权威**：客户端只发意图，服务端裁决并推状态；客户端最多做输入回显/按钮禁用/加载态。文字 MUD **不需要客户端预测**。
- **版本协商**：登录帧带协议版本+客户端版本+平台 → 服务端回 `VersionAccepted` / `UpgradeRequired`；协议不兼容直接拒，小版本靠 proto 新增字段+默认值兼容。

### proto 演进纪律
> **开发期（未上线）：proto 随便改、Dart/Rust 跟着重生成，不留任何兼容包袱（绿地原则 §0.1）。**
> 下列纪律**仅在上线、有不可控的已发布客户端后才生效**：
- 字段号一旦发布**永不复用**；删字段用 `reserved`（号+名都留）。
- 新增字段必须可选/有默认语义，**不靠字段顺序**。
- 弃用只标 `deprecated` + 保留解析，别让老客户端读出错。
- Rust 用 `prost`/`prost-build` 生成，Dart 用 `protoc` + `protoc-gen-dart`。

## 5. 持久化模型（PostgreSQL）

- **权威在内存**：游戏服把世界态（实体/房间/在线角色）放内存里跑，保证 tick 性能；PG 只做账号/角色/存档。
- **存档分级**（不许「随缘异步落库」）：
  - **即时事务落库**：账号注册、密码变更、创建角色、登录态、获得稀有资产、交易类（后续版本）。
  - **autosave**：普通属性/位置每 **30–60s** 批量落库脏数据。
  - **优雅关停**：SIGTERM → 停收新命令 → flush 全部脏角色 → 超时退出。
- **可接受丢失窗口**：可接受丢 ≤30s 普通进度；**绝不丢账号与建角**。
- **唯一性靠 DB 约束兜底**：账号名/角色名唯一约束，不靠应用层判断。
- **数据库 schema**：用声明式 `driver/schema.sql` 建库；**开发期直接 drop + 重建，不写任何数据迁移**（绿地原则 §0.1）。上线有真实玩家数据后才引入迁移工具。
- **单写入者**：MVP 单游戏服=唯一写入者，PG 留足并发读/后台/备份头量；将来多写入者无缝接住。

## 6. 脚本与热更（Lua）

- 状态在 Rust、逻辑在 Lua → 热更不丢档；`notify` 监听 `mud/scripts/` 与 `mud/data/`，改动即重载。
- 重载：加载新 chunk → 校验通过 → **原子切换**；失败**回滚到旧版本**，玩家无感。
- 所有 Lua 调用包 `pcall`/`Result`：脚本报错只记日志、不崩服。
- **沙箱红线**：禁 `io`/`os`/`debug`/动态 `require` 外部路径；禁全局可变状态、禁模块加载副作用、禁长期保存闭包、禁 Lua 自起 timer/thread；每次调用给 **CPU/指令预算**（`mlua` 内存上限 + 指令计数钩子）掐死死循环。

## 7. 数值与配置（弃 CSV）

| 类型 | 格式 | 例子 | 理由 |
|------|------|------|------|
| 声明式数值 | **TOML**（serde 校验） | 四维基础、天赋预设、门派一览、字段校验规则 | 带注释、diff 友好、**改错了启动即报错** |
| 带逻辑/公式 | **Lua** | 战斗公式、成长曲线、条件规则 | 直接进热更路径，表达力满 |

> 原 `planning/data/*.csv` 的定稿数值，落地时转成 `mud/data/*.toml`；带公式的部分迁到 Lua 脚本。

## 8. 安全

- 密码 argon2（blocking pool，不卡主循环）；传输 WSS（TLS）。
- 登录限流（失败计数，先内存即可）；同账号重复登录 → **踢旧连接**。
- 服务端权威 = 反作弊地基；所有校验在服务端。
- **GM/运维命令与玩家命令权限隔离**，别靠隐藏字符串。
- 日志对密码/token/原始敏感输入打码。

## 9. 可观测 & 可测试

- **指标**：在线数、命令耗时、Lua 调用耗时、DB 写耗时、channel 队列长度、autosave 失败计数。
- **可测试性**：游戏命令逻辑能**脱离网络**跑脚本级测试——这是 AI 大量代写 mudlib 的质量保险，第一天就建。

## 10. 工程结构（驱动 / 脚本分离 —— LPMud 式）

```
jiuzhoujianghu-lu/                  # 游戏仓库
├── driver/                         # 🔧 驱动开发目录（Rust 引擎源码, 只在这儿开发/编译）
│   ├── Cargo.toml                  # workspace
│   ├── crates/
│   │   ├── server/                 # bin: 主程序（tokio WS 接入 + 游戏主循环 + mlua + PG）
│   │   ├── protocol/               # lib: proto 生成的 Rust 类型 + 内部消息契约（接缝）
│   │   └── persistence/            # lib: sqlx(PG) 数据访问层（DAL，藏 SQL 细节）
│   ├── proto/                      # 🔑 .proto 契约（生 Rust; Flutter 端也引这里生 Dart）
│   ├── schema.sql                  # 声明式建库（开发期直接重建，无迁移）
│   └── deploy.sh                   # 编译 + 把二进制复制进 ../mud/
│
├── mud/                            # 🎮 脚本/运行目录（= 可整体复制的「一个服」单元）
│   ├── jzmud                       # ← 编译后的驱动二进制（复制进来；不入库）
│   ├── config.toml                 # 启动配置（端口/DB/路径/tick/服名…）
│   ├── scripts/                    # Lua mudlib（热更）
│   ├── data/                       # TOML 数值表（热更）
│   ├── logs/                       # 运行日志（不入库）
│   └── start.sh                    # 启动脚本（./jzmud --config config.toml）
│
├── planning/                       # 团队规划（gitignored, 独立仓库）
└── client/                         # （或独立仓库）Flutter 客户端, 引 driver/proto/ 生 Dart
```

- **驱动与脚本彻底分离**：引擎(Rust)只在 `driver/` 开发编译；`mud/` 是自包含、可整体复制的可运行实例。
- **入库的是游戏(scripts/data/config)，不是二进制**：`mud/jzmud`、`mud/logs/` 进 `.gitignore`。

## 11. 构建·部署·开服模型（FluffOS 式，一份驱动跑多服）

**核心理念**：驱动是通用引擎，靠 `config.toml` 决定一个服的全部行为；`mud/` 目录整体可复制 = 一个独立服务器。

```
开发驱动:  cd driver && cargo build --release
部署:      driver/deploy.sh   →  cp target/release/jzmud  ../mud/jzmud
启动:      cd mud && ./start.sh    (= ./jzmud --config config.toml)
```

`config.toml` 草案（驱动一切，改服只改它）：
```toml
server_name = "九州·测试一服"
listen      = "0.0.0.0:9000"      # WebSocket 监听
database_url = "postgres://user:pass@localhost/jzmud_s1"
scripts_dir = "./scripts"
data_dir    = "./data"
log_dir     = "./logs"
tick_ms     = 100
hot_reload  = true                # 监听 scripts/data 改动即热更
```

- **升级驱动**：重新 `cargo build` → 把新 `jzmud` 覆盖进 `mud/`（或各服目录）→ 重启。脚本/数据原样不动。
- **开新服**：`cp -r mud mud-2` → 改 `mud-2/config.toml`（端口/库名/服名）→ `cd mud-2 && ./start.sh`；或直接丢个新驱动二进制 + 改下启动脚本。
- **一份驱动、N 个 mud 实例**：多个 `mud*/` 共用同一编译产物，各自独立配置与数据库。

## 12. 分期演进路线（留好接缝，按压力付费）

| Phase | 加什么 | 触发条件 |
|-------|--------|----------|
| **1（MVP）** | 单 Rust 进程 + PG + Lua + WS/protobuf + Flutter | — |
| 2 | 拆独立 **Gateway** 进程（内部走 gRPC 或自定义传输） | 有活跃玩家、要重启 Rust 二进制不掉线 |
| 3 | **Redis** | 在线态/好友在线/推送扇出/限流/排行榜/缓存 |
| 4 | **NATS** | 多前端进程/后台服务/异步事件消费 |
| 远期 | 多游戏服**分片 + 跨服** | 单机吞吐真扛不住 |

> 接缝（`protocol` 消息契约 + `persistence` DAL + 主循环 channel 入口）让上面每步几乎零返工。
> **注意**：拆进程（Phase 2）必然要内部传输（gRPC/NATS）——「拆不拆进程」和「上不上 NATS」本质是同一个决定，故 MVP 单进程时两者都不需要。

## 13. 与首版 MVP 的关系

首版功能边界**不变**（注册 → 建角色 → 进江湖 → 看信息，见 [01-角色系统](01-角色系统.md)）。本架构是承载它的地基：MVP 即跑通「Flutter ⇄ WebSocket/protobuf ⇄ 单进程 Rust(主循环+Lua) ⇄ PostgreSQL」全链路，业务上只做角色相关四件事。门派及之后仍按路线图后置（YAGNI）。
