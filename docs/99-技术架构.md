## 0. 一句话

**Rust 驱动 + Lua 脚本热更**的现代化多人文字 MUD：**Flutter 客户端**经 **WebSocket 承载 protobuf** 连 **单进程 Rust 服务端**（tokio 管 I/O + 单线程游戏主循环独占世界态与 Lua VM），**PostgreSQL** 落库，数值走 **TOML + Lua** 双热更。**引擎与内容都做成「不可变版本化制品」，区服只是「指定用哪版引擎+哪版内容」的瘦实例**——开发、发布、开服、回滚全靠 config 指版本。性能、稳定优先，对 AI 友好，按需演进不提前付分布式税。

## 0.1 绿地原则（铁律，全程适用）

- **全新项目，零历史包袱**：不写任何向后兼容、不做任何数据迁移、不留旧格式适配层。
- 设计/实现一律**按当下最佳实践**；发现更优解就**直接重构/重写**，绝不为「兼容旧代码」将就。
- 开发期（未上线）：数据库 **drop + 重建**、proto 随改随生成、存档结构随便换——怎么干净怎么来。
- **唯一**需考虑兼容的情形：**上线后**面对不可控的已发布 Flutter 客户端时的 proto 向后兼容（见 §4）；未上线前一律不管。

## 1. 选型总表（定稿）

| 维度 | 选型 | 定位 |
|------|------|------|
| 驱动语言 | **Rust**（`tokio` 异步） | 内存安全 + 无 GC 停顿 + 海量长连接 |
| 脚本/热更 | **Lua**（`mlua`，lua54/vendored） | 命令、公式、行为，文件改了即热更 |
| 客户端 | **Flutter**（Android/iOS/桌面；web 可选） | 一套代码多端 GUI |
| 线格式 | **protobuf (proto3)** | 一份 `.proto` 生 Rust(`prost`)+Dart 两端类型 |
| 传输 | **WebSocket 承载 protobuf 二进制帧** | 长连接双向、穿透好、弱网重连简单、web 可复用 |
| 拓扑（MVP） | **单进程**（网关为模块，留接缝） | 见 §2；拆网关进 Phase 2 |
| 权威存储 | **PostgreSQL**（`sqlx`） | 账号、角色、持久数据；**每区独立库** |
| 数值-声明式 | **TOML + serde** | 四维/天赋/门派/校验规则；强类型校验当安全网 |
| 数值-公式 | **Lua** | 战斗、成长曲线、条件规则；进热更路径 |
| 密码 | **argon2** | 默认安全参数 + 随机 salt，禁明文 |
| 日志/错误 | `tracing` / `anyhow`+`thiserror` | 结构化日志，敏感字段打码 |
| 中间件 | **MVP 不上 NATS / Redis** | 按 §13 分期，触发了再加 |

## 2. 拓扑（MVP：单进程 + 接缝）

```
   Flutter 客户端 (Android/iOS/桌面)
            │  WebSocket（protobuf 二进制帧）
   ┌────────▼─────────────────────────────────────┐
   │       Rust 驱动（单进程 = 一个区服）              │
   │  ┌──────────────┐  channel   ┌───────────────┐  │
   │  │ 网关层(模块)    │ ─命令(in)─▶ │  游戏主循环      │  │
   │  │ WS·鉴权·会话·  │ ◀输出(out)─ │  (单线程, 独占    │  │
   │  │ protobuf 编解码 │            │  世界态 + Lua VM) │  │
   │  └──────────────┘            └───────┬───────┘  │
   │        ▲  tokio 多线程只管 I/O            │ 异步落库/读档 │
   └────────┼────────────────────────────────┼──────┘
            │                                 ▼
       连接保活/背压                    ┌─────────────┐
                                      │ PostgreSQL   │
                                      │ 账号·角色·存档 │
                                      └─────────────┘
```

- **「网关」当前是进程内模块**，与游戏主循环只通过 `protocol` 消息 + channel 对话，**互不直碰对方内存**——这就是「接缝」。Phase 2 拆进程时把它挪成独立二进制、channel 换网络传输即可，领域逻辑与消息契约原样搬。
- **核心原则（LPMud 思想）：状态住驱动（Rust/内存），行为住脚本（Lua）。** 热更换逻辑不丢世界态——这是热更能安全做的根基。

## 3. 运行时模型（最关键，别跑偏）

`mlua` 的 Lua VM **单线程**，`tokio` 多线程。二者这样接：

```
WS 连接(tokio 多线程, 只做 I/O) ──解码 protobuf→ bounded channel ──▶ 游戏主循环(单线程)
                                                                      │ 独占 World + Lua VM
        ◀── 编码 protobuf ── bounded channel ◀── 输出帧 ◀────────────┘
```

- **所有游戏逻辑 + Lua 调用全压到一个「游戏主循环」线程串行跑**：天然无数据竞争、确定性强、Lua 单线程刚好契合、热更安全。
- **tokio 只管异步 I/O**（WS 收发、DB I/O、定时器、密码哈希丢 blocking pool），**绝不阻塞主循环**；DB 读写走异步任务，结果投回主循环。
- **Lua 红线**：脚本里不许 await、不许阻塞 I/O、不许直接碰共享 Rust 状态；I/O 线程只投递命令，Lua 与世界态只在游戏线程触碰。
- **背压（强制断开策略）**：每连接进、出队列都设上限；**入站满 → 断该连接，出站满 → 断该连接**，绝不无限排队拖死主循环（MUD 最易死在慢客户端）。
- **tick 心跳**：固定 **100ms**（战斗等可另设更细），跑气血/内力恢复、NPC AI、定时事件、autosave；MVP 以事件驱动为主，tick 兜定时任务。Lua 通过 `heartbeat()`/定时器接口挂钩。

## 4. 客户端协议（Flutter + protobuf + WebSocket）

- **消息信封**：每帧一个 `Envelope { oneof payload; seq; ack; error?; request_id?; timestamp?; client_version? }`。单连接多路复用所有消息类型；**带错误包**（失败统一回 `error`）。
- **鉴权**：握手只建连接；**第一条业务帧 `AuthRequest`**（账号+token）→ 成功后绑定账号与角色。**绝不把 token 放 URL**（会进日志/代理/历史）。
- **心跳**：服务端 ping / 客户端 pong，**15–30s**，连续 **2–3 次**无响应才断；断的是连接，不立刻销毁角色。
- **断线重连状态机**：短断线（手机切前后台、WiFi↔流量）**保留 session 60–180s**；重连用 token **接管原 session**，宽限期内重挂回还在场的角色，**别每次都重走登录→进世界全流程**；超时后角色离世或进离线保护。
- **服务端权威**：客户端只发意图，服务端裁决并推状态；客户端最多做输入回显/按钮禁用/加载态。文字 MUD **不需要客户端预测**。
- **版本协商**：登录帧带协议版本+客户端版本+平台 → 服务端回 `VersionAccepted` / `UpgradeRequired`；协议不兼容直接拒，小版本靠 proto 新增字段+默认值兼容。

### proto 演进纪律
> **开发期（未上线）：proto 随便改、Dart/Rust 跟着重生成，不留任何兼容包袱（绿地原则 §0.1）。**
> 下列纪律**仅在上线、有不可控的已发布客户端后才生效**：
- 字段号一旦发布**永不复用**；删字段用 `reserved`（号+名都留）。
- 新增字段必须可选/有默认语义，**不靠字段顺序**。
- 弃用只标 `deprecated` + 保留解析，别让老客户端读出错。
- Rust 用 `prost`/`prost-build` 生成，Dart 用 `protoc` + `protoc-gen-dart`。

## 5. 持久化模型（PostgreSQL）

- **权威在内存**：游戏服把世界态（实体/房间/在线角色）放内存里跑，保证 tick 性能；PG 只做账号/角色/存档。
- **存档分级**（不许「随缘异步落库」）：
  - **即时事务落库**：账号注册、密码变更、创建角色、登录态、获得稀有资产、交易类（后续版本）。
  - **autosave**：普通属性/位置每 **30–60s** 批量落库脏数据。
  - **优雅关停**：SIGTERM → 停收新命令 → flush 全部脏角色 → 超时退出。
- **可接受丢失窗口**：可接受丢 ≤30s 普通进度；**绝不丢账号与建角**。
- **唯一性靠 DB 约束兜底**：账号名/角色名唯一索引、`characters.account_id UNIQUE`（一账号一角色），建角放事务里，冲突回「已有角色」。
- **数据库 schema**：声明式 `schema.sql` 建库；**开发期直接 drop + 重建，不写任何数据迁移**（绿地原则 §0.1）。驱动**启动时校验自己库的 schema 版本，不符直接拒绝启动**（守卫，非迁移）。上线有真实玩家数据后才引入迁移工具。
- **每区独立库**：各区服连各自的 PostgreSQL 库，互不干扰；单区服 = 该库唯一写入者。

## 6. 脚本与热更（Lua）

- 状态在 Rust、逻辑在 Lua → 热更不丢档。
- **dev**：`notify` 监听内容源 → **debounce 几十~几百 ms**（合并抖动、避开半写）→ 加载新 chunk → 校验 → **原子切换**；失败**回滚旧版本**，只记日志不崩服。
- **prod**：内容是不可变制品（见 §10），更新 = 切到新 content 版本（同样原子热切，或重启）。
- 所有 Lua 调用包 `pcall`/`Result`：脚本报错只记日志、不崩服。
- **沙箱红线**：禁 `io`/`os`/`debug`/动态 `require` 外部路径；禁全局可变状态、禁模块加载副作用、禁长期保存闭包、禁 Lua 自起 timer/thread；每次调用给 **CPU/指令预算**（`mlua` 内存上限 + 指令计数钩子）掐死死循环。
- **热更只替换脚本函数表，不动 Rust 世界态**；需要改世界态结构 = 重启区服（开发期 drop 重建），状态迁移不进 MVP。

## 7. 数值与配置（弃 CSV）

| 类型 | 格式 | 例子 | 理由 |
|------|------|------|------|
| 声明式数值 | **TOML**（serde 校验） | 四维基础、天赋预设、门派一览、字段校验规则 | 带注释、diff 友好、**改错了启动即报错** |
| 带逻辑/公式 | **Lua** | 战斗公式、成长曲线、条件规则 | 直接进热更路径，表达力满 |

> scripts 与 data 同属「内容」，一起冻结成不可变 content 版本（见 §10）。原 `planning/data/*.csv` 定稿数值落地转 `mudlib/data/*.toml`。

## 8. 安全

- 密码 argon2（默认安全参数 + 随机 salt，blocking pool，不卡主循环）；传输 WSS（TLS）。
- 登录限流（按账号/IP 内存失败计数）；同账号重复登录 → **踢旧连接**（MVP 单账号单在线）。
- 服务端权威 = 反作弊地基；所有校验在服务端。
- **GM/运维命令与玩家命令权限隔离**，别靠隐藏字符串。
- 日志对密码/token/原始敏感输入打码。**密钥（DB 密码等）不进 git**：仓库只放 `config.example.toml`，真配置在服务器（或走环境变量）。

## 9. 可观测 & 可测试

- **指标**：在线数、命令耗时、Lua 调用耗时、DB 写耗时、channel 队列长度、autosave 失败计数。
- **可测试性**：游戏命令逻辑能**脱离网络**跑脚本级测试——这是 AI 大量代写 mudlib 的质量保险，第一天就建。

## 10. 工程结构（引擎/内容/实例三层分离 · 不可变版本化制品）

**核心模型**：`driver`（引擎源）和 `mudlib`（内容源）是开发真相源 → 构建出**不可变版本化制品** `dist/engine/*` 与 `dist/content/*` → 区服是**只含 config 的瘦实例**，靠 config **钉死用哪版引擎 + 哪版内容**。

```
jiuzhoujianghu-lu/                  # 游戏仓库
├── driver/                         # 🔧 引擎源码（Rust workspace, dev 编译）
│   ├── crates/{server, protocol, persistence}
│   ├── proto/                      # .proto 契约（生 Rust; Flutter 也引这里生 Dart）
│   └── schema.sql                  # 声明式建库（开发期直接重建）
│
├── mudlib/                         # 📝 内容源码（dev 唯一真相源）
│   ├── scripts/                    # Lua（命令/公式/行为）
│   └── data/                       # TOML（声明式数值）
│
├── dist/                          # 📦 不可变制品库（构建产物, .gitignore）
│   ├── engine/jzmud-1.2.3          # 引擎各版本二进制（只读）
│   └── content/45/                 # 内容各版本快照（scripts+data 冻结只读）
│
├── servers/                        # 🎮 区服实例（瘦：只有 config + 运行态）
│   ├── s1/  { config.toml, logs/ }
│   └── s2/  { config.toml, logs/ }
│
├── config.example.toml             # 配置模板（入库；真配置/密钥不入库）
├── planning/                       # 团队规划（gitignored, 独立仓库）
└── client/                         # （或独立仓库）Flutter, 引 driver/proto/ 生 Dart
```

`servers/s1/config.toml`：
```toml
server_name  = "九州·一区"
listen       = "0.0.0.0:9001"
database_url = "postgres://user:pass@localhost/jzmud_s1"   # 每区独立库；密钥不入库
engine       = "dist/engine/jzmud-1.2.3"     # 用哪版引擎
content      = "dist/content/45"             # 用哪版内容（冻结不可变）
tick_ms      = 100
hot_reload   = false                          # prod 关；dev 指 live mudlib 时开
# 路径启动时转绝对（相对则相对 config 文件解析，绝不依赖进程 cwd）
```

**这套为什么最优（一次根除前几版所有坑）：**

| 性质 | 怎么来的 |
|------|---------|
| 无半写 / 无多文件中间态 | 制品**不可变**：发布 = 冻结成**新** `dist/content/N`，从不原地改已发布版本 |
| 无引擎/脚本版本 skew | config **同时钉死** engine + content 两个版本 |
| 无重复（胜过每区拷一份） | 同版本多区**共享同一份只读制品**——不可变所以共享安全，不拷 N 份 |
| 灰度白送 | 某区 config 指新版本即灰度；稳了再改其余区 |
| 开发极简 | dev 的 config `content = "mudlib"` + `hot_reload=true` → 改源码即热更，连发布都不用 |
| 实例极轻 | 区服目录只有 config + logs；开新区 = 建目录写 config |

> `.gitignore`：`dist/`、`servers/*/logs/`、真实 `config.toml` 不入库；入库的是源码 + `config.example.toml`。

## 11. 构建 · 发布 · 开服（本地/CI）

```
编译引擎:   cargo build --release        → dist/engine/jzmud-<ver>
冻结内容:   release.sh 把 mudlib/ 快照   → dist/content/<n>/   (不可变)
本地起一区: jzmud --config servers/s1/config.toml
```
- **开发**：config `content = "mudlib"` + `hot_reload=true`，改源码即热更，不发版。
- **发布内容**：冻结出 `dist/content/46` → 某区 config 切 46（热切或重启）→ 灰度 → 推全区。
- **升级引擎**：编出 `dist/engine/jzmud-1.3.0` → 区 config 切版本 → 重启（换二进制需重启）。
- **开新区**：建 `servers/sN/` + 写 config（端口/库/区名/版本）。**啥都不用拷**。

## 12. 部署到服务器（生产 · 服务器只放成品，干净）

**dev 仓库 = 造东西的工厂（源码+编译+冻结）；服务器 = 放成品的货架。服务器上没有源码、没有编译器、没有 git**——Rust 是单个二进制，Lua/TOML 是纯文本由引擎直接读、不在服务器编译。

服务器目录（就三个）：
```
/opt/jzmud/
├── engine/   jzmud-1.2.3  jzmud-1.3.0     # 引擎二进制（留几版备回滚）
├── content/  45/  46/                      # 冻结内容版本（只读）
└── servers/  s1/{config.toml,logs/}  s2/{…}
```

部署一条龙（`deploy.sh`，本地/CI 跑）：
```
1) Linux 容器/CI 编译: cargo build --release (目标 x86_64-unknown-linux-musl, 全静态)
2) release.sh 冻结 mudlib → content/<n>
3) rsync jzmud-<ver>  → 服务器 /opt/jzmud/engine/
   rsync content/<n>/ → 服务器 /opt/jzmud/content/<n>/
4) 改/下发该区 config（指向新版本）
5) systemctl restart jzmud@s1
```

systemd 模板（一个模板管所有区，开机自启 + 崩溃自拉起）：
```ini
# /etc/systemd/system/jzmud@.service
[Service]
WorkingDirectory=/opt/jzmud/servers/%i
ExecStart=/opt/jzmud/engine/jzmud-1.3.0 --config config.toml
Restart=always
[Install]
WantedBy=multi-user.target
```
```
systemctl enable --now jzmud@s1     # 起/开新区
```

- **更新内容**：rsync 新 `content/N` → 切 config 或热更，**不重新编译**。
- **升级引擎**：rsync 新二进制 → 切 config → `systemctl restart`（重启短暂掉线；**重启不掉线**留 Phase 2 网关）。
- **回滚**：旧版本制品都在，config 指回旧 `content`/旧二进制 → 重启，**秒回滚**。
- **静态编译**：`x86_64-unknown-linux-musl` 全静态（mlua vendored Lua C 一起静态进去），服务器丢一个文件就跑，不管 glibc 版本。服务器**不装任何工具链**（在 CI/Linux 容器编译）。
- **密钥**：服务器上的 `config.toml` 才有真 DB 密码（或环境变量），仓库只有 `config.example.toml`。

## 13. 分期演进路线（留好接缝，按压力付费）

| Phase | 加什么 | 触发条件 |
|-------|--------|----------|
| **1（MVP）** | 单进程 + PG + Lua + WS/protobuf + Flutter + 制品/区服模型 + systemd 部署 | — |
| 2 | 拆独立 **Gateway** 进程（内部走 gRPC 或自定义传输） | 引擎升级要**重启不掉线** / 多协议接入 |
| 3 | **Redis** | 在线态/好友在线/推送扇出/限流/排行榜/缓存 |
| 4 | **NATS** | 多前端进程/后台服务/异步事件消费 |
| 远期 | 单世界**分片 + 跨服** | 单区单机吞吐真扛不住 |

> 接缝（`protocol` 消息契约 + `persistence` DAL + 主循环 channel 入口）让上面每步几乎零返工。
> **注意**：拆进程（Phase 2）必然要内部传输（gRPC/NATS）——「拆不拆进程」和「上不上 NATS」本质同一个决定，故 MVP 单进程时两者都不需要。
> **区服 ≠ 分片**：区服是「同一/不同内容版本的多个独立世界（各自库）」，MVP 就支持；分片(远期)是「一个世界拆到多进程」，两码事。

## 14. 与首版 MVP 的关系

首版功能边界**不变**（注册 → 建角色 → 进江湖 → 看信息，见 [01-角色系统](01-角色系统.md)）。本架构是承载它的地基：MVP 即跑通「Flutter ⇄ WebSocket/protobuf ⇄ 单进程 Rust(主循环+Lua) ⇄ PostgreSQL」全链路，并以「不可变制品 + 瘦区服实例 + systemd」方式部署，业务上只做角色相关四件事。门派及之后仍按路线图后置（YAGNI）。
